## 2026-07-17 — „Flag weglassen" ≠ „lädt nichts": Digest-Self-Feed kam zurück
- Der 07-13-Fix stützte sich auf eine Annahme: `ClaudeAgentSdkProvider` OHNE `setting_sources` lade „keine Hooks". Falsch für claude-code 2.1.211 / SDK 0.2.120: bei weggelassenem `setting_sources` sendet das SDK gar kein `--setting-sources`, und die CLI lädt dann ihren EIGENEN user+project-Default (also Hooks) → Stop-Hook re-enqueued → ~1730 Digest-Prozesse in Minuten, Quota verbrannt.
- Regel: Annahmen über „lädt nichts" NIE aus einer Doku ableiten — gegen die installierte SDK-Version verifizieren (`inspect.getsource` auf den subprocess-Transport: wie mappt die Option auf CLI-Flags?). Leere Liste `setting_sources=[]` → `--setting-sources=` → lädt nichts; das ist der Unterschied zu None.
- Regel: Sicherheitskritische Invarianten defense-in-depth bauen, nicht auf einer einzelnen Annahme. Hier drei unabhängige Beine: (1) Env-Sentinel `REPO_AGENT_HARNESS_IN_DIGEST` macht `capture.enqueue` zum No-op im Digest-Kind (stärkstes, quellseitiges Bein — greift egal welche Hooks laden), (2) `setting_sources=[]` explizit, (3) Drain-Circuit-Breaker (Floor + Cap/min gegen Zero-Sleep-Spin). Regressionstest über echte Prozessgrenze (`subprocess` mit gesetztem Sentinel enqueued nichts).
- `docker/test.sh` direkt auf dem Host ausgeführt → dessen fehlgeschlagene `cd`s ließen `git add/commit` + `git config user.*` im echten Repo laufen (init-Commit mit t@t, lokale Identität überschrieben). Regel: Skripte mit Container-Pfaden (`/workspace`, `/plugins`) IMMER über ihren Runner (`docker/run.sh`) starten; vor Ausführung fremder Skripte die erwarteten Pfade prüfen.
- Folge-Regel: nach jedem fehlgeschlagenen Testlauf `git status` UND `git log -1` prüfen, bevor weitergearbeitet wird — der Fremd-Commit blieb sonst bis zur Ship-Phase unentdeckt.
